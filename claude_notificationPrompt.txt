🎯 MASTER PROMPT: CivicWatch Complete Notification System

OVERVIEW:
Build a comprehensive, real-time notification system for all four user roles in CivicWatch using Firebase Cloud Messaging (FCM). Notifications must be timely, relevant, actionable, and role-specific.

NOTIFICATION INFRASTRUCTURE:
Technology Stack:

Firebase Cloud Messaging (FCM) for push notifications
Cloud Functions for automated notification triggers
Firestore for notification history storage
Local Notifications for scheduled/reminder notifications

Notification Types:

Push Notifications - Real-time alerts (FCM)
In-App Notifications - Notification center inside app
Email Notifications - Optional (for important events)
SMS Notifications - Optional (for critical events)


NOTIFICATION ARCHITECTURE:
FCM Token Management:
On User Login:

Request FCM token from device
Store in Firestore: users/{uid}.fcmToken: "token_string"
Update token on every login (tokens can change)

On User Logout:

Optionally clear FCM token OR keep for logged-out notifications

Token Refresh:

Listen for token refresh events
Update Firestore with new token


Notification Storage (Firestore):
Collection: notifications
Document ID: auto-generated
{
  id: string
  userId: string (recipient UID)
  userRole: 'admin' | 'state_authority' | 'city_authority' | 'citizen'
  type: string (notification type enum)
  title: string
  body: string
  data: object (additional payload)
  read: boolean (default: false)
  actionUrl: string (deep link to relevant screen)
  createdAt: timestamp
  expiresAt: timestamp (optional, for auto-delete)
}
Indexes Required:

userId + read + createdAt (for unread notifications query)
userId + createdAt (for notification list)

7. CLOUD FUNCTIONS FOR AUTOMATED NOTIFICATIONS
Function 1: sendNotification
Purpose: Generic function to send FCM notification
Trigger: Called by other functions or manually
Input:
javascript{
  userId: string,
  title: string,
  body: string,
  data: object,
  type: string
}
Logic:

Fetch user's FCM token from Firestore users/{userId}.fcmToken
Check user's notification preferences (if notification type is disabled, skip)
Check quiet hours (if enabled and current time is in quiet hours, skip unless critical)
Send FCM notification via Firebase Admin SDK
Store notification in Firestore notifications collection
Return success/failure


Function 2: onIssueCreated
Trigger: Firestore onCreate for issues/{issueId}
Logic:

Get issue city
Query all City Authorities where city == issue.city and isActive == true
For each City Authority:

Call sendNotification with type: 'new_issue'


Check if any citizens have "nearby issues" enabled
Query citizens within 500m radius (using Firestore geohash or external service)
Send "nearby issue" notifications


Function 3: onIssueStatusUpdated
Trigger: Firestore onUpdate for issues/{issueId} (when status field changes)
Logic:

Get old status and new status
Get issue reporter (citizen UID)
Send notification to reporter with type: 'status_updated'
If new status == 'Done':

Send special notification with type: 'issue_resolved' (mentions before/after photos)


Get all users who upvoted/commented (follower list)
Send notification to followers with type: 'status_updated_followed_issue'


Function 4: checkEscalations (Scheduled)
Trigger: Cloud Scheduler (runs daily at 12:00 AM)
Logic:

Query all issues where status == 'Recognized' AND daysSinceLastUpdate >= 7 AND escalated == false
For each issue:

Update Firestore: issues/{issueId} → escalated: true, escalatedAt: timestamp, escalationReason: 'status_unchanged_7_days'
Get State Authority UID for that state
Send notification to State Authority with type: 'issue_escalated'
Get City Authority UID assigned to issue
Send notification to City Authority with type: 'issue_escalated_warning'
Send notification to issue reporter (citizen) with type: 'issue_escalated'


Query all issues where status == 'In Work' AND daysSinceLastUpdate >= 14 AND escalated == false
Repeat same logic as step 2


Function 5: sendEscalationWarnings (Scheduled)
Trigger: Cloud Scheduler (runs daily at 9:00 AM)
Logic:

Query issues where status == 'Recognized' AND daysSinceLastUpdate == 5 (2 days before escalation)
For each issue:

Get assigned City Authority
Send notification with type: 'escalation_warning', daysRemaining: 2


Query issues where status == 'In Work' AND daysSinceLastUpdate == 12
Repeat step 2


Function 6: onCommentAdded
Trigger: Firestore onCreate for comments/{commentId} (subcollection under issues/{issueId})
Logic:

Get issue ID from parent collection
Get issue details (reporter UID, follower list)
Get commenter UID and role
If commenter is City Authority:

Send to reporter + followers with type: 'official_update'


If commenter is State Authority:

Send to assigned City Authority with type: 'sa_comment'


If commenter is Citizen:

Send to reporter (if different) with type: 'comment_on_my_issue'
Send to followers with type: 'comment_on_followed_issue'




Function 7: onUpvoteMilestone
Trigger: Firestore onUpdate for issues/{issueId} (when upvotes field changes)
Logic:

Get old upvote count and new upvote count
Check if milestone crossed (10, 25, 50, 100)
If milestone crossed:

Send to issue reporter with type: 'my_issue_upvote_milestone'
Send to assigned City Authority with type: 'upvote_milestone'




Function 8: sendDailySummaries (Scheduled)
Trigger: Cloud Scheduler (runs daily at 9:00 AM)
Logic:

Calculate yesterday's stats:

Issues reported (total, by state, by city)
Issues resolved
Issues escalated
Pending approvals (CA registrations)


Send to Admin with type: 'daily_summary', stats: {...}
For each State Authority:

Calculate stats for their state
Send with type: 'daily_summary'


For each City Authority (if enabled in preferences):

Calculate stats for their city
Send with type: 'daily_summary'




Function 9: sendWeeklySummaries (Scheduled)
Trigger: Cloud Scheduler (runs every Sunday at 6:00 PM)
Logic:

Calculate last 7 days stats for each city
For each citizen (if enabled in preferences):

Get their city
Send summary with type: 'weekly_summary'

```
## **9. DEEP LINKING & NAVIGATION**

**When User Taps Notification:**

**Foreground (App Open):**
- Navigate to relevant screen immediately

**Background/Closed (App Not Open):**
- Open app → Parse `data.actionUrl` → Navigate to screen

**Deep Link Format:**
```
civicwatch://notifications/{notificationId}
civicwatch://issues/{issueId}
civicwatch://dashboard
civicwatch://pending-requests
civicwatch://escalated-issues
```

**Navigation Logic:**
```
data.actionUrl = "/issues/12345"
→ Navigate to IssueDetailScreen(issueId: "12345")

data.actionUrl = "/dashboard"
→ Navigate to Dashboard

data.actionUrl = "/pending-requests"
→ Navigate to PendingRequestsScreen
